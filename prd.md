## 1. 프로젝트 개요

### 프로젝트명
**팩토스튜디오 (FACTO Studio)**

### 한 줄 설명
산업용 로봇팔, 컨베이어, 카메라 비전을 PLC 카드 방식으로 직관적으로 제어하는 교육용 자동화 체험 플랫폼.

### 배경 및 필요성
- **산업 자동화 교육의 진입 장벽**: 실제 PLC 프로그래밍(래더 로직, ST 언어 등)은 초·중등 학생이 접근하기 어렵다. 산업 현장의 자동화 원리를 체험하려면 전문 지식과 고가 장비가 필요하다.
- **실물 하드웨어 체험의 가치**: 시뮬레이션만으로는 "실제로 물건이 움직이는" 감각을 제공하기 어렵다. 로봇팔이 물건을 집고, 컨베이어가 돌아가고, 카메라가 색상을 인식하는 실물 경험이 학습 효과를 높인다.
- **카드 기반 UI의 교육 효과**: 복잡한 코드 대신 동작 카드를 순서대로 배치하는 방식은 초등학생도 "순차 실행", "조건 분기", "반복" 등 프로그래밍 개념을 자연스럽게 익힐 수 있게 한다.
- **PLC 일관 커리큘럼**: 카드 UI(Phase 1)로 PLC 개념을 체험한 후, IEC 61131-3 ST 언어(Phase 2)로 자연스럽게 전환하여 실제 산업용 PLC 프로그래밍까지 연결되는 일관된 학습 경로를 제공한다.

## 2. 목표 및 비전

### 핵심 목표
학생들이 **실제 산업용 장비를 안전하게 제어**하면서 자동화의 원리(순차 실행, 센서 기반 분기, 반복)를 체험하고, 단계적으로 **실제 PLC 프로그래밍 역량**을 갖출 수 있도록 한다.

### 전체 학습 경로

```
Phase 1 (카드 방식)                         Phase 2 (ST 언어 방식)
─────────────────────────────────────       ──────────────────────────────────────
1~4차시:  순차 실행 (입력+출력)      ──▶    Phase 2a: ST 블록 에디터
5~8차시:  조건/분기 + 타이머         ──▶    (드래그&드롭 ST 함수 블록 조립)
9~12차시: 변수/데이터                ──▶    Phase 2b: ST 텍스트 에디터
13~16차시: 서브루틴 (FB 개념)        ──▶    (코드 직접 작성·수정·실행)
                                            Phase 2c: SFC/LD 뷰 + 내보내기
                                            ──▶ 실제 산업용 PLC (Codesys, TIA Portal)
```

### Phase 1 목표 — PLC 카드 기반 UI
- 초등 저학년~중학생까지 대상별로 활용 가능한 **카드 기반 자동화 플로우** 구성.
- **PLC 프로그램 방식 반영**: 조건 분기(IF/ELSE), 반복(LOOP/WHILE), 병렬 실행, 연속 스캔 사이클.
- 컨베이어 구동, 센서 감지, 카메라 색상 인식, 로봇팔 동작을 **42종 카드**로 제어한다.
- **블록 구조 편집기**: 카드 들여쓰기, 블록 연결선, 자동 페어링으로 구조적 프로그래밍 지원.
- 하드웨어 없이도 **시뮬레이션 모드**로 학습 가능하다.
- **E-STOP(비상정지)** 등 안전 장치가 항상 우선 동작한다.
- **단계별 해금**: 교사가 수업 차시에 맞게 카드 카테고리 노출 범위를 설정한다.
  - 1~4차시 (초등 저학년): 흐름 제어 + 입력 + 출력 (순차 실행만)
  - 5~8차시 (초등 고학년): + 조건/분기 + 타이머/카운터
  - 9~12차시 (초등 고학년 심화): + 변수/데이터
  - 13~16차시 (중학생): + 서브루틴 (전체 해금)

### Phase 2 목표 — IEC 61131-3 ST 블록 코딩
- 중학생 이상(13~16세)이 **IEC 61131-3 ST(구조화 텍스트) 기반 블록 코딩**으로 더 복잡한 자동화 로직을 구성할 수 있다.
- PLC 산업 표준 언어(ST)를 교육용으로 재해석하여 **카드 → ST 블록 → ST 텍스트 → 실제 PLC**라는 연속적 학습 경로를 제공한다.
- 변수 선언(VAR), 함수 블록(FUNCTION_BLOCK), 복합 조건, 타이머 등 **산업 PLC 프로그래밍 개념을 직접** 학습한다.
- Phase 1과 동일한 Flow Engine을 재사용하여 **동일한 하드웨어를 제어**한다.
- **하드웨어 독립성**: ST 문법은 IEC 국제 표준이므로, 로봇/컨베이어를 다른 제품으로 교체해도 학생이 작성한 ST 코드는 그대로 유효하다.

### 성공 지표 (KPI)

| 지표 | 목표값 | 측정 방법 |
|------|--------|-----------|
| 첫 플로우 완성 시간 | 5분 이내 | 학생이 처음 사용 시 카드 3장 이상의 플로우를 완성하는 데 걸리는 시간 (1~4차시 기준) |
| 독립 미션 완수율 | 80% 이상 | 교사 도움 없이 해당 차시 내장 미션을 완수하는 학생 비율 |
| E-STOP 반응시간 | 500ms 이내 | 비상정지 버튼 클릭부터 모든 장비 정지까지 소요 시간 |
| 시스템 안정성 | 연속 2시간 | 오류 없이 연속 수업 운영 가능 시간 |
| 카드→ST 전환율 | 60% 이상 | Phase 1 수료 후 Phase 2 ST 블록 코딩으로 전환하는 학생 비율 |
| ST 코드 자력 작성율 | 50% 이상 | Phase 2b 수료 후 카드 없이 ST 코드만으로 미션을 완수하는 학생 비율 |

## 3. 타겟 사용자

### Phase 1 주요 사용자

#### 학생 (초등 고학년, 10~13세)
- **배경**: 컴퓨터 기본 조작 가능, 코딩 경험 없거나 스크래치 수준
- **동기**: "로봇팔이 실제로 움직이는 걸 보고 싶다", "내가 만든 프로그램으로 기계를 제어하고 싶다"
- **특성**: 시각적 피드백에 민감, 긴 텍스트보다 아이콘/그림 선호, 시행착오 학습 선호

#### 지도교사
- **배경**: 정보/과학/기술 교과 담당, SW 교육 기본 역량 보유
- **동기**: 학생들에게 실물 자동화 체험 기회 제공, 수업 설계 도구 필요
- **특성**: 수업 전 미션 준비, 수업 중 안전 관리, 학생 진행상황 파악 필요

### Phase 2 추가 사용자

#### 학생 (중학생 이상, 13~16세)
- **배경**: Phase 1 카드 경험으로 순차/분기/반복 개념 습득 완료
- **동기**: PLC 산업 표준 언어(ST)로 더 복잡한 자동화 로직 구성, 산업 현장 기술과의 연결
- **특성**: 텍스트 기반 사고 시작, 변수/함수 개념 학습 가능, ST 구문이 카드 블록 구조와 1:1 대응되어 자연스러운 전환

### 사용자 페르소나

**민수 (초등 6학년, 12세)**
> "나는 레고 로봇을 좋아하는데, 진짜 공장 로봇은 어떻게 움직이는지 궁금해요. 카드를 놓으면 로봇이 움직인다니 신기해요!"

- 스크래치로 간단한 게임을 만들어 본 경험
- 마우스 조작에 익숙, 키보드 타이핑은 느림
- 실패해도 다시 시도하는 편, 즉각적인 피드백을 좋아함

**지훈 (중학교 2학년, 14세)**
> "카드로 로봇을 움직였는데, 진짜 공장에서 쓰는 코드가 어떻게 생겼는지 궁금해요. 나중에 자동화 엔지니어가 되고 싶어요."

- Phase 1 수료, 프로그래밍 기초 개념 이해
- 텍스트 코드 입력 가능, 산업 현장에 관심
- "이게 진짜 PLC 코드랑 같은 거예요?" 라는 질문을 자주 함

**김 선생님 (초등 정보 교사)**
> "SW 교육에 로봇팔 체험을 넣고 싶은데, 안전이 제일 걱정이에요. 미리 세팅해두면 학생들이 알아서 해볼 수 있으면 좋겠어요."

- 방과후 교실에서 8~12명 동시 수업 운영
- 하드웨어 설치/연결은 가능하나 깊은 프로그래밍은 어려움
- 학생 안전이 최우선, 수업 시간(40분) 내 완결되는 미션 필요

### 사용 환경
- **장소**: 학교 방과후 교실, 메이커 스페이스, 과학관 체험존
- **PC**: Windows 10/11 또는 macOS (Electron 데스크톱 앱)
- **네트워크**: 교실 WiFi (ESP32-CAM 스트리밍용), 인터넷 필수 아님
- **하드웨어 세팅**: 교사가 수업 전 설치·연결, 학생은 소프트웨어만 조작
- **수업 시간**: 1회 40~50분 (설명 10분 + 실습 30분)

## 4. 사용자 시나리오 (User Stories)

### 학생 시나리오

**US-1: 첫 자동화 플로우 만들기**
> 민수는 '시작' 카드를 캔버스에 놓고, '컨베이어 켜기' 카드를 아래에 추가한다. 'IR 감지 대기' 카드를 놓으면 센서가 물체를 감지할 때까지 기다린다. '컨베이어 끄기' 카드를 추가하고 실행 버튼을 누른다. 화면에 "실행할까요?" 확인 창이 뜨고, "예"를 누르면 컨베이어가 돌아가기 시작한다. 물체가 센서 앞에 도착하면 컨베이어가 멈춘다. 민수는 "내가 만든 프로그램으로 진짜 기계가 움직였다!"며 기뻐한다.

**US-2: 색상별 분류 미션**
> 민수는 내장 미션 "택배 분류"를 선택한다. 카드 플로우를 구성한다: 컨베이어 켜기 → IR 감지 대기 → 컨베이어 끄기 → 카메라 촬영 → 만약 빨간색이면 → 로봇 이동(A) → 물건 집기 → 로봇 이동(B) → 물건 놓기. 실행하면 빨간 물체가 B 위치로 분류된다. 파란 물체를 C 위치로 분류하는 조건을 추가해본다.

**US-3: 반복 동작 프로그래밍**
> 민수는 '반복하기' 카드를 사용하여 "컨베이어 켜기 → 감지 대기 → 끄기 → 집기 → 놓기"를 5번 반복하는 플로우를 만든다. 여러 물체가 연속으로 처리되는 걸 보며 "자동화가 이런 거구나" 하고 이해한다.

**US-4: 시뮬레이션으로 미리 테스트**
> 하드웨어가 연결되지 않은 상태에서, 민수는 시뮬레이션 모드를 켜고 플로우를 실행한다. 화면에 각 카드가 순서대로 실행되는 것이 표시되고, 가상의 센서 이벤트와 카메라 결과가 시뮬레이션된다. 실제 장비에 연결하기 전에 로직이 맞는지 확인한다.

**US-5: ST 코드 변환 확인 (Phase 2 진입)**
> 지훈이는 Phase 1에서 만든 택배 분류 플로우를 열고 ST 에디터 탭으로 전환한다. 카드와 1:1 대응되는 ST 코드가 실시간으로 생성되어 표시된다. `IF color_result = 'RED' THEN` 코드가 '만약 빨간색이면' 카드와 하이라이트로 연결된다. 지훈이는 "아, 이게 진짜 PLC 코드구나"라고 이해한다. 코드에서 `'RED'`를 `'BLUE'`로 수정하고 실행하니 파란 물체가 분류된다.

### 교사 시나리오

**US-6: 수업 준비 — 단계별 해금 설정**
> 김 선생님은 수업 전 프로그램을 실행하고 수업 프로파일에서 "5~8차시 (조건/분기 + 타이머)"를 선택한다. 학생 화면에서는 ①흐름 제어, ②입력, ③출력, ④조건/분기, ⑤타이머/카운터 카테고리만 활성화된다. 나머지 카테고리는 흐리게 표시된다. 설정을 `.bridge.json`으로 저장하여 학생들에게 배포한다.

**US-7: 비상 상황 대응**
> 수업 중 학생의 로봇팔이 예상과 다른 방향으로 움직인다. 김 선생님은 화면 우측 상단의 빨간 비상정지 버튼을 즉시 누른다. 모든 장비가 즉시 정지하고, 화면에 "비상정지 작동됨" 메시지가 표시된다. 안전을 확인한 후 플로우를 수정하고 다시 실행한다.

**US-8: PLC 뱃지 모드로 개념 연결 설명**
> 김 선생님은 헤더의 "PLC 용어 보기" 토글을 켠다. 각 카드에 PLC 대응 용어 뱃지가 표시된다. 'N초 기다리기' 카드 옆에 `TON`, '횟수 세기' 카드 옆에 `CTU` 뱃지가 붙는다. "이게 실제 공장에서 TON이라고 불러요"라고 설명하며 카드와 PLC 개념을 연결한다.

**US-9: 플로우 저장 및 공유**
> 김 선생님은 수업용 예제 플로우를 만들어 `.bridge.json` 파일로 저장한다. 학생들에게 파일을 배포하면 학생들은 "불러오기"로 예제를 열어 수정하고 실행해볼 수 있다.

#### FR-1: 팩토블록 팔레트 — 이중 분류 체계

화면 왼쪽에 팩토블록 팔레트를 표시한다. 팔레트는 **프로그래밍 개념 중심 7개 카테고리**로 구성되며, 각 카드에는 **하드웨어 장치 태그 뱃지**를 병기하는 이중 분류 체계를 사용한다.

> **v3.3 변경**: v3.1의 하드웨어 장치 중심 5개 카테고리(기본/컨베이어/센서/비전/로봇)에서 프로그래밍 개념 중심 7개 카테고리로 전환. 기존 카드 기능은 유지하되 팔레트 분류 재구성. 하드웨어 장치 분류는 카드 태그 뱃지로 제공.

##### FR-1-A: 7개 카테고리 구성

|------|-----------|------|--------------|----------|
| ① | 흐름 제어 | 회색 `#6B7280` | OB·태스크 + FOR/WHILE | 1차시~ |
| ② | 입력 (센서 읽기) | 청록 `#0D9488` | `%I` 입력 태그 | 1차시~ |
| ③ | 출력 (장치 제어) | 파랑 `#3B82F6` | `%Q` 출력 태그 | 1차시~ |

##### FR-1-B: 카드 목록 (이중 분류 체계 반영)

각 카드는 **팔레트 카테고리(분류 위치)**와 **하드웨어 태그(뱃지 표시)**를 함께 가진다.

**① 흐름 제어 카테고리** — 프로그램 구조 + 반복/병렬

| 카드 ID | 한글명 | 구조 타입 | 설명 |
|---------|--------|----------|------|
| `start` | 시작 | marker | 플로우의 시작 지점 |
| `stop` | 끝 | marker | 플로우 종료 |
| `loop` | 반복하기 | block_open | N회 단순 반복 블록 (횟수 파라미터, 카운터 변수 없음) |
| `end_loop` | 반복하기 끝 | block_close | LOOP 블록 종료 |
| `while_cond` | ~하는 동안 | block_open | 조건부 반복 블록 시작 |
| `end_while` | 반복 끝 | block_close | WHILE 블록 종료 |
| `parallel_start` | 동시에 시작 | block_open | 병렬 블록 시작 |
| `parallel_branch` | 작업 나누기 | block_middle | 병렬 갈래 구분 |
| `parallel_end` | 모두 완료되면 | block_close | 모든 갈래 완료 후 진행 |

> **`loop` vs `loop_n` 차이**: `loop`(① 흐름 제어)는 단순 N회 반복으로 카운터 변수에 접근 불가. `loop_n`(⑤ 타이머/카운터)는 FOR 루프로 카운터 변수를 블록 내부에서 활용 가능 (예: "i번째 물체를 i번 위치로 이동"). 교육 단계에 따라 순차적으로 해금.

| `capture` | 카메라 촬영 | action | `[카메라]` | 카메라 캡처 |

#### FR-1-C: 단계별 블록 해금 정책 *(신규)*

교사가 수업 차시에 맞게 노출 블록 범위를 설정할 수 있다. 해금 설정은 수업 프로파일로 저장·배포된다.

##### 단계별 해금 로드맵

| 차시 | 대상 | 해금 카테고리 | 핵심 학습 목표 | 대표 미션 |
|------|------|--------------|--------------|---------|
| 1~4차시 | 초등 저학년 | ① + ② + ③ | 순차 실행 체험 | 컨베이어 켜고 끄기, IR 감지 후 정지 |
| 5~8차시 | 초등 고학년 | + ④ + ⑤ | 조건 분기 + 시간 제어 체험 | 색상별 택배 분류, N초 자동 정지 |
| 9~12차시 | 초등 고학년 심화 | + ⑥ | 데이터 저장·활용 체험 | 불량품 수량 누적 세기 |
| 13~16차시 | 중학생 이상 | + ⑦ (전체) | 재사용 로직 설계 체험 | 분류 서브루틴 정의 후 재활용 |

##### 해금 세부 요건

- **FR-1-C-1**: 교사 화면에서 수업 프로파일(1~4 / 5~8 / 9~12 / 13~16차시)을 선택하면 해당 단계의 카테고리만 활성화된다.
- **FR-1-C-2**: 잠긴 카테고리는 팔레트에서 흐리게(opacity 40%) 표시되며, 클릭 시 "선생님이 아직 잠가 두었어요" 안내가 표시된다.
- **FR-1-C-3**: 수업 프로파일은 `.bridge.json`의 `unlockProfile` 필드에 저장되며, 파일 배포 시 학생 화면에도 동일 설정이 적용된다.
- **FR-1-C-4**: 교사는 특정 카테고리를 개별 ON/OFF 할 수 있어 로드맵과 다른 커스텀 조합도 가능하다.
- **FR-1-C-5 (PLC 뱃지 모드)**: 헤더에 "PLC 용어 보기" 토글을 제공한다. ON 시 각 카드에 PLC 대응 용어 뱃지(예: `TON`, `CTU`, `%Q`)가 표시된다. 교사 시연 시 PLC 개념 연결 설명에 활용한다. 학생 화면 기본값은 OFF.

#### FR-2: 플로우 캔버스 (팩토조립소)

- 화면 중앙에 플로우 캔버스를 표시한다.
- 카드 팔레트에서 캔버스로 **드래그&드롭**하여 카드를 추가한다.
- 캔버스 내에서 카드 **순서를 드래그로 변경**할 수 있다.
- **블록 구조 시각화**: IF/LOOP/WHILE/PARALLEL 블록 내 카드를 들여쓰기로 표시.
- **블록 연결선**: 블록 시작~끝 카드를 좌측 세로선으로 연결.
- **자동 페어링**: block_open 카드 배치 시 대응하는 block_close 카드 자동 삽입.
- **블록 단위 이동**: block_open 드래그 시 블록 전체(내부 카드 포함) 이동.
- **유효성 검증**: 짝이 맞지 않는 블록 구조 시 경고 표시.
- 캔버스 카드를 **드래그로 삭제**할 수 있다: ① 팩토블록(팔레트)으로 드래그 드롭 ② 화면 하단에 나타나는 휴지통(TrashZone)으로 드래그 드롭.
- 현재 실행 중인 카드를 **하이라이트**로 표시한다 (currentCardUid 기반).
- 캔버스가 비어있을 때 안내 텍스트를 표시한다: "왼쪽에서 카드를 끌어다 놓으세요!"

#### FR-3: 카드 파라미터 편집

- 파라미터가 있는 카드를 클릭하면 편집 패널을 표시한다.
- 파라미터 종류별 적절한 입력 UI를 제공한다:
  - 숫자 (delay, loop): 슬라이더 + 숫자 입력
  - 선택 (robot_move position): 드롭다운 (A/B/C)
  - 선택 (if_sensor sensor): 드롭다운 (센서1/센서2)
  - 선택 (conv_speed level): 슬라이더 (1~5)
  - 변수 참조 (counter_up, var_set): 변수 이름 입력 또는 드롭다운
- 파라미터 범위를 벗어난 값은 입력을 차단한다.

#### FR-4: 플로우 실행 제어

- **실행** 버튼: 캔버스의 카드 플로우를 순차 실행한다.
- **일시정지** 버튼: 현재 카드 완료 후 대기 (실행 중에만 표시).
- **재개** 버튼: 일시정지 상태에서 다음 카드부터 실행 (일시정지 중에만 표시).
- **정지** 버튼: 실행을 완전히 중단하고 IDLE 상태로 복귀.
- 실행 전 확인 대화상자를 반드시 표시한다: "이 플로우를 실행할까요?"
- 현재 상태(IDLE/RUNNING/PAUSED/WAITING/ERROR)를 화면에 표시한다.

#### FR-5: E-STOP 비상정지

- 화면 우측 상단에 항상 보이는 **빨간색 원형 비상정지 버튼**을 배치한다.
- 클릭 즉시 WebSocket으로 `e_stop` 메시지를 전송한다.
- Python 백엔드가 모든 하드웨어(로봇/컨베이어/ESP32)를 즉시 정지시킨다.
- 비상정지 후 "비상정지 작동됨" 메시지와 함께 복구 안내를 표시한다.
- E-STOP은 어떤 상태에서든 (RUNNING/PAUSED/WAITING/ERROR) 항상 동작해야 한다.

#### FR-6: 하드웨어 연결 관리

- 연결 패널에서 3종 장비의 연결을 관리한다:
  - **로봇팔** (UFactory Lite 6): IP 주소 입력 + **그리퍼 타입 선택 (표준/진공)** → 연결/해제
  - **ESP32** (컨베이어): COM 포트 선택 (자동 탐지) → 연결/해제
  - **카메라** (ESP32-CAM): IP 주소 입력 → 연결/해제 (사무실 192.168.1.xxx / 현장 192.168.10.xx)
- 각 장비의 연결 상태를 색상 아이콘으로 표시한다:
  - 초록: 연결됨, 파랑: 시뮬레이션, 빨강: 오류, 회색: 미연결
- 연결 실패 시 오류 메시지와 재시도 버튼을 제공한다.
- **로봇 연결 시** 위치 교시 패널(PositionEditor)이 하단에 표시된다.

#### FR-7: 카메라 실시간 뷰어

- ESP32-CAM의 MJPEG 스트림을 실시간으로 표시한다.
- 색상 감지 결과를 오버레이로 표시한다 (감지된 색상, 신뢰도).
- 카메라 미연결 시 "카메라 연결 필요" 안내를 표시한다.

#### FR-8: 하드웨어 상태 모니터

- 화면 하단 상태바에 장비 연결 상태를 표시한다.
- 로봇팔, ESP32, 카메라 각각의 상태(연결/해제/오류)를 실시간 업데이트한다.
- 오류 발생 시 상태바에 오류 메시지를 표시한다.

#### FR-9: 플로우 저장/불러오기

- 현재 플로우를 `.bridge.json` 파일로 저장한다.
- 저장 내용: 카드 목록, 파라미터, 로봇 위치, 하드웨어 설정, 수업 프로파일(unlockProfile).
- `.bridge.json` 파일을 불러와 플로우를 복원한다.
- 내장 미션 예제 4종을 제공한다:
  1. **벨트 제어 기초** (1~4차시): 컨베이어 켜기→IR 감지 대기→끄기
  2. **센서 활용** (5~8차시): 컨베이어 켜기→감지 대기→정지→로봇 이동
  3. **택배 분류** (5~8차시): 컨베이어→감지→카메라→색상 분기→로봇 분류
  4. **불량품 카운터** (9~12차시): 변수+카운터+조건 복합 활용

#### FR-9-A: `.bridge.json` 버전 관리 정책 *(신규)*

#### FR-10: 시뮬레이션 모드

- 하드웨어 없이 소프트웨어만으로 플로우를 테스트할 수 있다.
- 시뮬레이션 모드에서는 각 카드의 실행을 시간 지연으로 대체한다.
- 센서 이벤트, 카메라 결과를 가상 데이터로 시뮬레이션한다.
- 화면에 "시뮬레이션 모드" 표시를 명확히 한다.

#### FR-10a: 로봇 위치 교시

- 로봇 연결 시 우측 패널에 **위치 교시 패널**을 표시한다.
- HOME, A, B, C 4개 위치의 좌표(X/Y/Z/Roll/Pitch/Yaw)를 편집할 수 있다.
- **"현재 위치 읽기"** 버튼으로 로봇의 실제 TCP 위치를 읽어온다.
- 읽어온 좌표를 **"적용" → "저장"** 순서로 해당 위치에 등록한다.
- 저장된 위치는 safety_manager에 즉시 반영되어 카드 실행 시 사용된다.
- `.bridge.json` 저장 시 교시된 위치도 함께 저장된다.

#### FR-10b: 로봇 Live Control (실시간 수동 제어)

- 로봇 연결 시 **Live Control 패널**을 제공한다.
- **카테시안 조그**: X/Y/Z/Roll/Pitch/Yaw 각 축의 +/- 버튼으로 로봇을 수동 이동한다.
- **속도 슬라이더**: 조그 이동 속도를 10~300mm/s 범위에서 조절한다.
- **Teaching 모드 토글**: 로봇을 자유 이동 상태로 전환하여 손으로 잡고 교시할 수 있다.
- E-STOP은 Live Control 중에도 항상 동작한다.

#### FR-10c: GPIO 모니터/제어

- 로봇 연결 시 **GPIO 모니터 패널**을 제공한다.
- **디지털 입력** (CI0~CI7, 24V): 실시간 상태를 LED로 표시한다.
- **디지털 출력** (CO0~CO7, 24V NPN): 클릭으로 ON/OFF 토글 제어한다.
- **아날로그 입력** (AI0~AI1, 0~5V): 전압 값을 표시한다.
- **Tool GPIO** (IO0, IO1, 5V): 상태 표시 및 클릭 토글 제어한다.
- 500ms 주기로 I/O 상태를 폴링하여 UI를 갱신한다.

## 6. 비기능 요구사항

### NFR-1: 성능
- WebSocket 메시지 왕복 시간: **< 100ms** (localhost 환경)
- 카메라 MJPEG 스트림 지연: **< 500ms**
- 카드 드래그&드롭 반응: **< 50ms** (60fps 이상)
- 플로우 저장/불러오기: **< 1초**
- 카드→ST 코드 실시간 변환: **< 200ms**

### NFR-2: 안전
- E-STOP 반응시간: 버튼 클릭→장비 정지 **< 500ms**
- 비상정지는 프로그램 상태와 무관하게 항상 동작
- 로봇팔 교육 모드 속도 제한: **100mm/s** 이하
- 로봇팔 충돌 감도: **레벨 3/5** (민감)
- 실행 전 사용자 확인 대화상자 필수

### NFR-3: 신뢰성
- Serial 통신 타임아웃: **3초**, 재시도: **2회**
- WebSocket 연결 끊김 시 자동 재연결 시도
- 하드웨어 통신 실패 시 안전하게 정지 (fail-safe)
- 연속 **2시간** 이상 오류 없이 운영 가능

### NFR-4: 사용성
- 모든 UI 텍스트는 **한글** (ST 코드 키워드 제외)
- 전문 용어 대신 쉬운 표현 사용 (예: "컨베이어 시작" → "컨베이어 켜기")
- 카드에 직관적 아이콘(이모지) 사용
- 폰트 크기 최소 **14px** (대상 연령 고려)
- 첫 사용 시 **5분 이내**에 첫 플로우 완성 가능

### NFR-5: 확장성
- Flow Engine은 `FlowCard[]` 배열을 입력받는 범용 구조
- Phase 2 ST 블록/텍스트 코딩에서 ST → FlowCard[] 변환으로 동일한 Engine 재사용 가능
- 새로운 카드 타입 추가 시 `CardDefinition` + ST 함수 블록 등록으로 확장 가능
- 새로운 하드웨어 추가 시 Controller 클래스 추가로 확장 (ST 코드 변경 불필요)
- IEC 61131-3 국제 표준 기반이므로 타 PLC 플랫폼(Codesys, TIA Portal)과 코드 호환 가능

## 8. 하드웨어 인터페이스

### 8.1 UFactory Lite 6 로봇팔
- **통신**: TCP/IP (xArm-Python-SDK), 노트북 ↔ 로봇 LAN 직결 `192.168.1.xxx`
- **교육 모드 설정**: 최대 속도 100mm/s, 최대 가속도 500mm/s², 충돌 감도 3/5
- **동작 범위**: 사전 정의 위치(HOME/A/B/C)만 허용, UI에서 교시 가능
- **그리퍼 (2종)**: 표준(Lite6 내장) + 진공, UI 연결 시 라디오 버튼 선택
- **Live Control**: Mode 0 Incremental 조그 + Mode 2 Teaching
- **GPIO**: CGPIO 8DI+8DO(24V) + AI 2ch + AO 2ch + TGPIO 2IO(5V)
- **전원**: 별도 AC 전원 (자체 어댑터)

### 8.2 ESP32 + 스텝모터 컨베이어
- **통신**: USB Serial (pyserial, 115200bps)
- **컨트롤러**: ESP32 DevKit V1 + TB6600 스텝모터 드라이버
- **프로토콜**: JSON + 개행, 타임아웃 3초, 재시도 2회
- **명령어**: `CONV_START`, `CONV_STOP`, `CONV_SPEED`

### 8.3 ESP32-CAM
- **스트리밍**: HTTP MJPEG (`http://{ip}:81/stream`)
- **캡처**: HTTP GET (`http://{ip}/capture`) → JPEG
- **색상 분석**: Python OpenCV로 캡처 이미지 처리
- **WiFi**: 교실 로컬 네트워크 연결

### 8.4 E18-D80NK 적외선 센서 (x2)
- **타입**: NPN 출력 (감지 시 LOW)
- **연결**: ESP32 GPIO 18, GPIO 19 (INPUT_PULLUP)
- **전원**: 5V (3.3V 불가), 배선: 갈색=VCC, 파랑=GND, 검정=OUT

### 8.5 전원 설계
- **메인**: 민웰 RS-100-24 SMPS (24V 4.5A, 108W)
- **24V 경로**: RS-100-24 → TB6600 → 스텝모터 (피크 ~48W)
- **5V 경로**: LM2596 스텝다운 → ESP32 + ESP32-CAM + IR센서 (~5W)
- **Lite 6**: 별도 AC 전원 (메인 SMPS와 분리)
- **총 소비**: 피크 ~53W / 용량 108W = 49%

4. **크기가 중요도** — 액션 카드 > 조건 카드 > 닫기 카드 순으로 크기 차등
5. **일러스트가 주인공** — 텍스트보다 그림이 먼저 눈에 들어옴

### 11.4 ST 코드 실행 안전 제한
- ST 코드에서 호출 가능한 함수는 사전 정의된 하드웨어 제어 함수로만 제한
- 임의 시스템 명령, 파일 접근, 네트워크 호출은 ST 파서에서 차단
- 로봇 위치 명령은 교시된 위치(HOME/A/B/C)만 허용

### 11.5 오류 시 안전 정지
- 하드웨어 통신 오류 → 모든 장비 정지 (fail-safe)
- 사용자에게 오류 원인 및 조치 방법 안내

## 13. 리스크 및 제약사항

### 하드웨어 의존성
- **리스크**: 실물 하드웨어 없이 전체 기능 테스트 불가
- **완화**: 시뮬레이션 모드(FR-10) + Hardware Controller Mock

### ST 파서 복잡도
- **리스크**: IEC 61131-3 ST 전체 스펙 구현은 복잡
- **완화**: FR-17 정의 범위(부분 집합)만 구현, 미지원 문법은 명확한 오류 메시지

### 네트워크 환경
- **리스크**: 교실 WiFi 불안정 → ESP32-CAM 스트리밍 품질 저하
- **완화**: MJPEG 해상도/프레임 조절, 카메라 외 통신은 USB Serial

### 안전 사고
- **리스크**: 로봇팔 동작 중 학생 접촉
- **완화**: 교육 모드 100mm/s, 충돌 감도 상향, E-STOP, 사전 정의 위치만 허용, ST 안전 제한(11.4)

### Electron + Python 프로세스 관리
- **리스크**: Python 비정상 종료 시 포트 점유
- **완화**: `python-manager.ts` 상태 감시, 앱 종료 시 kill, 포트 중복 검사

### 크로스 플랫폼 호환성
- **리스크**: COM 포트 이름 체계 OS별 차이
- **완화**: 포트 자동 탐지 + 수동 선택

## 14. 용어 정의 (Glossary)

| 용어 | 설명 |
|------|------|
| **PLC** | Programmable Logic Controller. 본 프로젝트에서는 카드 UI(Phase 1)와 ST 에디터(Phase 2)로 PLC 개념과 프로그래밍을 단계적으로 교육. |
| **팩토스튜디오 (FACTO Studio)** | 본 프로젝트의 전체 제품명. Electron 데스크톱 앱. |
| **팩토블록 (FACTO Block)** | 카드 팔레트의 브랜드 명칭. 7개 카테고리 카드 팔레트. |
| **팩토조립소 (FACTO Build Station)** | 중앙 편집 영역. 카드 플로우 캔버스 + ST 에디터. |
| **이중 분류 체계** | 프로그래밍 개념 카테고리(팔레트)와 하드웨어 태그(뱃지)를 함께 사용하는 분류 방식. |
| **단계별 해금 (Unlock)** | 수업 차시에 따라 블록 카테고리를 순차 활성화. 교사가 수업 프로파일로 설정. |
| **수업 프로파일** | `.bridge.json`에 저장되는 수업 설정 (해금 단계, HW 연결 정보, 로봇 위치). |
| **PLC 뱃지 모드** | 카드에 PLC 대응 용어(`TON`, `CTU`, `%Q`) 뱃지를 표시하는 교사용 기능. |
| **IEC 61131-3** | PLC 프로그래밍 언어 국제 표준. ST, LD, FBD, SFC, IL 5개 언어 포함. |
| **ST (Structured Text)** | IEC 61131-3의 텍스트 기반 언어. 파스칼 유사 문법. Phase 2 핵심. |
| **SFC (Sequential Function Chart)** | IEC 61131-3의 상태 전이도 언어. Phase 2c 뷰어. |
| **LD (Ladder Diagram)** | IEC 61131-3의 릴레이 회로 시각 언어. Phase 2c 뷰어. |
| **카드 (Card)** | 자동화 동작 하나를 추상화한 UI 요소. |
| **플로우 (Flow)** | 카드를 순서대로 배치한 자동화 프로그램. |
| **Flow Engine** | 플로우(카드 배열 또는 ST 코드)를 실행하는 Python 백엔드 엔진. |
| **E-STOP** | Emergency Stop. 모든 하드웨어를 즉시 정지하는 안전 장치. |
| **ESP32** | WiFi/BLE 마이크로컨트롤러. 컨베이어 제어 및 카메라에 사용. |
| **ESP32-CAM** | 카메라 모듈 통합 ESP32. MJPEG 스트리밍 가능. |
| **xArm SDK** | UFactory 로봇팔 제어 Python SDK. |
| **Lite 6** | UFactory 6축 협동 로봇팔. |
| **Monaco Editor** | VS Code 기반 코드 에디터 라이브러리. Phase 2 ST 에디터에 사용. |
| **Zustand** | React 상태 관리 라이브러리. |
| **@dnd-kit** | React 드래그&드롭 라이브러리. |
| **시뮬레이션 모드** | 하드웨어 없이 소프트웨어만으로 플로우를 테스트하는 모드. |
